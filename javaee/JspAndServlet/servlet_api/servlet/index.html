    Servlet 规范

    Web 应用的部署文件结构规范

    WEB-INF：这个文件夹的名称是固定的，而且一定位于项目的根目录之下。WEB-INF 文件夹对外界是封闭的，浏览器无法直接访问其资源，必须通过 Servlet/Jsp 的请求转发。不想让浏览器直接存取的资源，可以放在这个文件夹下。

    • web.xml：Web 应用的部署描述文件。
    • lib：放依赖的Jar包
    • classes：编译过后的 .class 文件。

    other：例如 images、css 等静态资源文件

    /WEB-INF/lib 中 Jar 包的 /META-INF/resources 中的静态资源也会被访问到，比如 /index.html ，如果根目录下不存在 index.html，那么会使用 Jar 包中的 /META-INF/resources/index.html（如果存在的话）。

    Servlet API

    Servlet API架构图

    Servlet 应用程序，编译时候必须要有 Servlet API 相关的类，如果用的是 Tomcat 的话，这些类在 Tomcat 文件夹的 lib 子文件夹中 servlet-api.jar 包中。

    Servlet 生命周期

    init(Servletconfig servletConfig)：实例化的方法， Servletconfig 对象会被当作参数传入。

    service()：servlet 业务处理的方法。

    destroy()：Servlet 实例销毁前的回调。

    ServletConfig

    Servlet 实例的配置信息，可以在声明 Servlet 的时候对其进行设置。

    声明 Servlet

    @WebServlet

    @WebServlet{
    name="hello",
    urlPattern={"/hello"},
    initParams={
        @WebInitParam(name="path", value="/xxx"),
        @WebInitParam(name="name", value="ruby")
    },
    loadOnStartup=1
}

    Web.xml

    <servlet>
    <servlet-name>hello</servlet-name>
    <servlet-class>com.example.Hello</servlet-class>
    <load-on-startup>1</load-on-startup>
    <init-param>
        <param-name>path</param-name>
        <param-value>/xxx</param-value>
    </init-param>
    <init-param>
        <param-name>name</param-name>
        <param-value>ruby</param-value>
    </init-param>
</servlet>
<servlet-mapping>
    <servlet-name>hello</servlet-name>
    <url-pattern>/hello</url-pattern>
</servlet-mapping>

    属性

    name：servlet 实例的名称

    urlPatterns：servlet 的路径匹配，可以设置多个。

    initParams：Servlet 配置，会被映射到 ServletConfig 中。

    loadOnStartup：

    • 如果值大于0，则表示 Web 容器启动后就初始化相应的 Servlet 实例，初始化的顺序由值大小决定，值越小优先级越高，值相同则由 Web 容器厂商自行决定。

    • 如果值小于0，则表示 Web 容器启动后不会立马实例化相应的 Servlet 实例，而是会在首次接收请求时候才对相应的 Servlet 进行实例化、初始化操作。

    • 默认值是-1

    GenericServlet

    GenericServlet 是一个抽象类，同时实现了 Servlet 和 ServletConfig 接口。

    GenericServlet 实现了 Servlet#init(ServletConfig) 方法，并且添加了空的无参 init 方法。

    public void init(ServletConfig config) throws ServletException {
    this.config=config;
    init();
}
//我们在实现 Servlet 的时候，如果需要初始化资源，可以重写该方法，而不是带参数的 init 方法。
public void init(){};

    GenericServlet 由于同时实现了 ServletConfig 接口，所以可以直接调用下面的 api

    • ServletConfig getServletConfig()
    • String getInitParameter(String name)
    • Enumeration getInitParameterNames()
    • ServletContext getServletContext()

    HttpServlet

    HttpServlet 继承于 GenericServlet，同时定义 Web Http 相关的处理 API，具体可以查看开头的架构图。GenericServlet 在最初定义 Servlet 的时候，并不限于它只能用于 Http，所有并没有把 HTTP 相关服务流程定义在 GenericServlet 中，而是定义在 HttpServlet 的 service() 方法中。与 Servlet 相关的类或者接口都位于 javax.servlet 包中，如：Servlet、GenericServlet、ServletRequest、ServletResponse 等。而与 Http 定义相关的类或者接口都位于 javax.servlet.http 包中，如：HttpServlet、HttpServletRequest、HttpServletResponse 等。

    ServletContext

    ServletContext 是 Servlet 应用程序运行环境的上下文，其定义了一些 API 可以实现获取请求资源的 URI、存取属性，设置或者获取应用程序的初始参数，甚至动态设置 Servlet 实例。

    getRequestDispatcher(String path)：获取 RequestDispatcher 对象，path 参数必须以 ”” 开头。

    "/" 开头的路径成为环境相对路径，没有 “/” 开头的路径称为请求相对路径。

    getResourcePaths(String path)：列出路径目录下的所有文件路径，路径全部为展开的环境相对路径，同样 path 参数必须是以 “/” 开头。

    getResourceAsStream(String path)：返回文件的输入流，path 必须以 “/” 开头，表示环境相对路径，或者依赖 jar 包下 META-INF/resources 的相对路径。

    PushBuilder

    Servlet 4.0 后提出的新 API，用于支持 Http 2.0 的 Server Push。

    ServletRequest

    HttpServletRequest

    处理请求参数：getParameter()、getParameterValues()、getParameterNames()、getParameterMap()

    处理请求头：getHeader()、getHeaders()、getHeaderNames()

    Post 请求参数的编码处理：

    • 对每个 ServletRequest，调用其 setCharacterEncoding(String encoding) 方法。
    • 在 web.xml 中配置，例如：<request-character-encoding>utf-8</request-character-encoding>。
    • 在 Filter 中，统一对 ServletRequest 对象，调用 setCharacterEncoding 方法设置。

    Get 请求参数处理：

    • setCharacterEncoding 方法，只会对请求体起作用，Get 方法参数存在 URI 中，因此不会起作用，因为处理 URI 的是 HTTP 服务器，不是 WEB 服务器，因此需要手动转换。例如：string name=new String(name.getBytes("ISO-8859-1"),"UTF-8")。
    • Tomcat 8 之后，处理 URI 的默认编码是 UTF-8，但在 Tomcat 7 之前是 ISO-8859-1，对于 Tomcat 7 的 GET 编码处理可以参考 Filter 使用请求封装器统一处理

    直接读取请求 Body 输入流：

    • getReader()：可以取得一个 BufferedReader，读取 Body 的数据。
    • getInputStream()：可以取得一个 ServletInputStream，这是 InputStream 的子类，可以直接读取 Body 的串流，可以利用它俩处理上传文件的区段。

    multipart/form-data 处理：getPart()、getParts()，Part 接口的具体使用参考这里

    ServletResponse

    HttpServletResponse

    设置响应头：

    • setHeader()、addHeader()
    • setXXXHeader()、addXXXHeader()，XXX 表示类型，可以是 Int、String、Date等。

    设置缓冲区：

    • getBufferSize()：获取缓冲区大小。
    • setBufferSize()：设置缓存区大小，必须在 HttpServletResponse 的 getWriter() 和 getOutputStream() 方法被调用之前进行调用，否则会抛出 IllegalStateException 异常。
    • isCommitted()：在缓冲区未满之前，设置的响应内容并不会真正传至浏览器，可以使用 isCommitted() 方法确认是否已经响应确认。
    • reset()：重置响应内容，会连同标头一起清除，如果响应已经确认，调用该方法会抛出 IllegalStateException 异常。
    • resetBuffer()：重置响应内容，但是不会清除标头，如果响应已经确认，调用该方法会抛出 IllegalStateException 异常。
    • flushBuffer() ：清除缓冲区以设置的响应信息，并且传送至浏览器。

    HttpServetResponse 对象若被关闭，则必须清除所有的响应内容，响应对象被关闭的时机由以下几个：

    • Servlet 的 service() 方法已经结束。
    • 响应内容的长度超过了 HttpServletResponse#setContentLength() 所设置的长度。
    • 调用了 sendRedirect() 或者 sendError() 方法。
    • 调用了 AsyncContext 的 complete() 方法。

    设置输出编码 （优先级高到低）：

    • web.xml 中配置，例如：<response-character-encoding>UTF-8</response-character-encoding>。
    • 通过 HttpServletResponse#setCharacterEncoding 和 HttpServletResponse#setContentType() 方法。

    setContentType 方法可以设置 MIME 和 编码：setContentType("text/html;charaset=utf-8")

    常见的 MIME 设置有：text/html, application/pdf, application/json 等。可以在 web.xml 中设置文件后缀名以 MIME 的映射，这样可以通过 ServletContext#getMimeType(String filename) 获取对应的 MIME。

    <mime-mapping>
    <extension>pdf</extension>
 <mime-type>application/pdf</mime-type>
</mime-mapping>
    • 通过 HttpServletResponse#setLocale 来设置地区信息，locale 包含了语系和编码信息。

    可以通过 web.xml 来自定义 locale 的编码信息

    <locale-encoding-mapping-list>
    <locale-encoding-mapping>
    <locale>zh_CN</locale>
     <encoding>UTF-8</encoding>
 </locale-encoding-mapping>
</locale-encoding-mapping-list>

    输出字符：getWriter()

    输出二进制流：getOutputStream()，

    重定向：sendRedirect()

    发送错误编码：sendError()，如果在响应确认之后调用会抛出 IllegalStateException 异常。
