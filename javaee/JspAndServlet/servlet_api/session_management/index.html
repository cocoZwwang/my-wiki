    会话管理

    会话管理的几个基本方式：隐藏域（Hidden Field）、Cookie 与 URI 重写。

    隐藏域

    把前面页面的结果通过隐藏域的方式写入新的页面，然后返回给浏览器，浏览器下一次提交的时候，就会把隐藏域中的内容一并提交，从而实现状态的简单管理。比如一个调查问卷，在用户提交第一页答案准备跳转到第二页的时候，后端可以把第一页的答案写入第二页的隐藏域，当用户需要从第二页返回第一页的时候，后端可以从隐藏域中取出第一页的作答情况，从而保留了用户第一页的作答页面。这是一种非常简单的状态管理，并不算 Servlet/JSP 的管理机制。

    Cookie

    Cookie 的基本原理

    Cookie 是由浏览器负责存储信息的一种方式，Web 应用程序可以通过请求头 set-cookie 来告诉浏览器需要存储的信息的内容和期限。例如：

    set-cookie user=ruby;Expires=SUN,12-Apr-2009 04:38:55 GMT

    Web 端设置请求头set-cookie 的方式如下：

    Cookie cookie = new Cookie("user","ruby");
cookie.setMaxAge(7*24*60*60)//有效期为7天，如不设定有效期限，则默认关闭浏览器后失效
response.addCookie(cookie);//HttpServletResponse 可以 add 多个 cookie 对象

    Web 应用程序可以通过 HttpServletRequest#getCookies() 来获取网页所属域（domain）的所有 Cookie，返回值是一个 Cookie[]。

    Cookie[] cookies=request.getCookies();
if(cookies!=null){
    for(Cookie cookie : cookies){
        String name=cookie.getName();
        String value=cookie.getValue();
    }
}

    浏览器被预期能为每个网站存储 20 个 Cookie，总共可存储 300 个 Cookie，每个 Cookie 大小不超过 4 KB（每个厂商的浏览器实际情况会有所不同），因此 Cookie 可存储的信息是有限的。

    Cookie 的安全性

    Cookie 把信息存储在客户端，被窃取的风险较高，虽然可以通过设置适当的过期时间，并且结合摘要算法在一定程度上提高安全性，但是还是无法保证，因此不太适合存储安全性要求较高的信息，比如密码。

    Cookie 可以通过 setSecure() 方法设定为 true，这样就只会在 HTTPS 下传送 Cookie。

    Servlet 3.0 之后可以通过 setHttpOnly() 方法在 set-cookie 头上附加 HttpOnly 属性，在浏览器支持的情况下，这个 Cookie 将不会被 javascript 读取。

    URI 重写

    所谓 URI 重写，其实就是 GET 请求参数的应用，当 Web 应用程序响应浏览器上一次请求时，将某些相关信息以超链接的方式响应给浏览器，这些超链接中包括请求参数信息。比如分页按钮，可以在响应时，为每个按钮的超链接附加 start 和 count 参数，这样就可以知道每一页查询数据的范围。

    因为 GET 本身可以携带的参数长度有限，因此只适合用来做一些简单的浏览器信息保留，或者辅助会话管理。

    HttpSession

    上面提到的三种方式不同，本质上都是由客户端来进行信息存储，可储信息的量有限，安全性也不高，因此只能用来做一些简单的状态管理，而 HTTP Session 则是由 Web 应用负责存储信息的一种方式。

    HttpSession 本质上是一个 JVM 中的对象，类似于 Map，以键值对的方式存储信息：

    HttpSession#setAttribute(String name, Object value)：设置属性。

    HttpSession#getAttribute(String name)：取出属性。

    HttpSession 并非线程安全，多线程环境中必须注意属性设定时共享存储的问题。

    会话管理：HttpSession + Cookie

    单纯靠 HttpSession 是无法完成会话管理的，一个 Web 应用每天可能会有很多浏览器的请求，HttpSession 无法靠自己识别自身所存储的信息分别对应于那个浏览器，只有浏览器携带了可识别其身份的标识才可以，这个标识是 Session ID。

    HttpSession 默认使用 Cookie 来辅助管理 Session ID 的状态，HttpSession 创建的时候，通过 set-cookie 来设置 Session ID，通常 name 为 JSESSIONID， value 为一串按一定规则生成的随机字符串：

    set-cookie JESSIONID=1C58EFDA8C4.....

    浏览器下一次请求的时候，便会把 Session ID 通过 cookie 头发送给 Web 应用，Web 应用根据 Session ID，便可以取出该浏览器之前存储的属性，从而实现会话状态的管理。

    HttpSession 有效期设置

    • HttpSession#setMaxInactiveInterval(long second)
    • ServletContext#setSessionTimeout(int minute)

    <web-app>
    <session-config>
        <!-- 单位为 minute -->
        <session-timeout>30</session-timeout>
    </session-config>
</web-app>

    Session ID 的有效期

    Session ID 默认是由 Cookie 来负责存储的，通过上面 Cookie 章节的介绍，我们知道如果手动在 set-cookie 的时候设定有效期，那么该 cookie 的有效期默认到浏览器关闭，这样下一次用户再打开浏览器访问之前的网站时候，便无法使用之前存储的 HttpSesssion 信息。

    但是 Session ID 的 set-cookie 是由 HttpSession 创建时候自动完成的，不需要我们手动添加，所以我们无法简单通过 Cookie#setMaxAge(int second) 方法来完成有效期的配置。

    不过 Servlet 给我们提供了 SessionCookieConfig 接口，我们可以通过其来设置 Session ID 对应 Cookie 的属性：

    // SessionCookieConfig 必须在 ServletContext 初始化之前完成设置。
// 一个方法是 ServletContextListener#contextInitialized() 中使用这些 API
SessionCookieConfig config = servletContext.getSessionCookieConfig();
config.setName("mysessionId");//设置 Session ID Cookie name，默认是 JSESSIONID
config.setMaxAge(7*24*60*60);// 设置过期时间
config.setSecure(true);// 只通过 Https 传送该 Cookie
config.setHttpOnly(true);// 不允许 javascript 读取该 cookie

    ServletContextListener 参考

    我们也可以在 Web.xml 中配置：

    <cookie-config>
    <name>mysessionId</name>
    <max-age>604800</max-age>
    <secure>true</secure>
    <http-only>true</http-only>
</cookie-config>

    单纯使用 HttpSession + Cookie 同样还是不够安全的，建议一般不采用 JSESSIONID 作为 Session ID 名称，并且使用 Https 进行加密传输，用户登录成功后要变更 Session ID，并且必要时候不要只通过 HttpSession 中的信息来判断用户身份。如果是重要的操作步骤，最好还需要额外的身份确认，比如支付步骤前需要再次输入支付密码。

    会话管理：HttpSession + URI 重写

    HttpSession 默认使用 Cookie 来存储 Session ID，但是如果用户关闭了浏览器的 Cookie 的功能，在这种情况下如果还想使用 HttpSession 来进行会话管理，可以考虑使用 URI 重写。

    URI 重写就是在响应的浏览器的页面的时候，对页面中的超链接的 URI 附加 Session ID，例如：

    http://localhost/hello?name=ruby;jessionid=23DF2AC24

    可以使用 HttpServletResponse#encodeURL() 方法来协助 URI 重写，例如：

    PrintWriter out=response.getWriter();
out.printf("<a href='%s'>点击就送</a>%n",response.encodeURL("hello?name=ruby"));

    如果 Web 应用能从浏览器的请求中取出 Session ID 对应的 Cookie，则 URI 不变，否则在后面附加 ;jsessionid=xxxx。

    URI 重写会把 session ID 暴露在 URI 中，因此更加不安全，这和用户可能是为了安全而关闭 cookie 的初衷是相反的。
